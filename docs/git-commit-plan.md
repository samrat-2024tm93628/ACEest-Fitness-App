# Git Commit & Branch Plan — Assignment 2

This is the recommended order for committing all Phase A–G work to the
repository. The plan produces a **clean linear history on `main`** plus
**six PRs** that demonstrate the pull-request workflow (which is graded in
its own right).

> Use this if you have NOT yet committed any of the Phase A–G changes.
> Run the commands top-to-bottom from the repository root.

---

## Branching strategy

| Branch | Purpose | Merges into |
|---|---|---|
| `main` | Production-ready, tagged releases only | — |

| `feature/a2-phase-a-hygiene` | Phase A: pinned deps, env config | `main` (PR #1) |
| `feature/a2-phase-b-container` | Phase B: multi-stage Dockerfile + compose | `main` (PR #2) |
| `feature/a2-phase-c-quality` | Phase C: coverage + smoke tests | `main` (PR #3) |
| `feature/a2-phase-d-cicd` | Phase D: Sonar + Jenkins + GHA | `main` (PR #4) |
| `feature/a2-phase-e-k8s` | Phase E: Kubernetes manifests + scripts | `main` (PR #5) |
| `feature/a2-phase-g-docs` | Phase G: docs + report | `main` (PR #6) |

### Merge policy

| Branch type | Push directly to `main`? | Merge style |
|---|---|---|
| `main` | **No** — protected. Merges only via PR. | — |
| `feature/*` | **No** — push to remote, open PR | **Squash-merge** (one commit per phase on `main`) |
| `bug/*` | **No** — open PR, must pass CI | Squash-merge |
| `hotfix/*` | Allowed by maintainer only, with tag | Fast-forward + tag |

> **Why squash-merge?** Each phase becomes one commit on `main` with a
> clean message — the `main` log reads like a release log. Detailed
> per-file commits remain visible in the PR for grading.

### Tag after final merge

After all 6 PRs are merged, tag the release: `v4.0.0-devops-a2`.

## Step 0 — Sanity check before starting

```bash
git status                              # confirm you see the modified + untracked files
git checkout main
git pull origin main                    # sync with remote
git stash --include-untracked           # park current changes safely
git log --oneline -5                    # confirm clean baseline
```

After each phase below, you'll `git stash pop`, stage **only that phase's
files**, commit, and stash the rest again.

> ⚠️ Do NOT run `git add -A` or `git add .` — it will pull files from
> later phases into the wrong commit. Always stage **explicit paths**.

## Cheat sheet — common commands

| Goal | Command |
|---|---|
| See files in current change set | `git status` |
| See what a path will do | `git diff -- path/to/file` |
| Stage one explicit path | `git add path/to/file` |
| Undo last `git add` | `git restore --staged path/` |
| List branches | `git branch -a` |
| Delete merged branch (local + remote) | `git branch -d <br>; git push origin --delete <br>` |
| Create PR from CLI | `gh pr create --base main --title "..." --body "..."` |
| List your PRs | `gh pr list --author @me` |
| Merge a PR (squash) | `gh pr merge <num> --squash --delete-branch` |

## Alternative — single squashed commit (only if time-pressured)

If you have **no time** for six PRs, the minimal acceptable commit history is:

```bash
git checkout -b feature/a2-full-devops
git add <all the files>
git commit -m "feat: Assignment 2 — full DevOps pipeline (phases A-G)

See docs/report.md for full description."
git push -u origin feature/a2-full-devops
gh pr create --base main --title "Assignment 2 — full DevOps pipeline" --body "See docs/report.md"
```

This loses the per-phase narrative on `main` but still demonstrates the
PR workflow. **Prefer the six-PR plan if time permits.**
